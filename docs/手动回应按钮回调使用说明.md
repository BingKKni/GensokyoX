# 手动回应按钮回调功能使用说明

## 📄 功能概述

本功能用于**手动回应按钮回调事件**（INTERACTION_CREATE），可以自定义点击按钮后弹出的提示文字（成功 / 失败 / 没有权限 等），替代默认的自动"操作成功"提示。

框架提供 **两条独立的调用路径**：

| 路径 | 适用场景 | 接口 |
|---|---|---|
| **A. 独立接口** | 只想回应按钮，不发消息 | `POST /put_interaction` |
| **B. 与发消息合并** | 既要回应按钮，又要发消息（一次 API 完成） | `send_group_msg` / `send_private_msg` 携带 `interaction_id` |

## 🔢 回应代码（`code`）说明

| 代码 | 含义 | 用户看到的提示 |
|------|------|---------------|
| 0 | 成功 | 操作成功（默认） |
| 1 | 操作失败 | 操作失败 |
| 2 | 操作频繁 | 操作过于频繁 |
| 3 | 重复操作 | 已重复操作 |
| 4 | 没有权限 | 没有权限 |
| 5 | 仅管理员操作 | 仅管理员可操作 |

---

## 路径 A：独立 HTTP API `/put_interaction`

适用于"**只回应按钮、不发实际消息**"的纯净场景，或回应与消息发送解耦的复杂业务流。

### 入参（GET / POST 均可）

| 字段 | 类型 | 说明 |
|---|---|---|
| `interaction_id` | string | **平台原始 `d.id`**，不做任何转换。与 `message_id` 二选一 |
| `message_id` | string | **框架转换后的数字短 ID**，自动通过 `idmap` / `memory_msgid` 还原为真实 `d.id`。与 `interaction_id` 二选一 |
| `code` | int | 回应代码 0~5，默认 `0`（成功） |

> **优先级**：同时提供 `interaction_id` 与 `message_id` 时，**`interaction_id` 优先**，`message_id` 被忽略。
> **必填**：两者至少提供其一，否则返回 400。

### 调用示例

**JSON POST**：
```bash
curl -X POST http://<host>:<port>/put_interaction \
  -H "Content-Type: application/json" \
  -d '{"interaction_id":"1a2b3c4d5e6f7890","code":4}'
```

**GET**：
```
GET /put_interaction?interaction_id=1a2b3c4d5e6f7890&code=4
```

**用框架短 msg_id**（自动还原为真实 `d.id`）：
```bash
curl -X POST http://<host>:<port>/put_interaction \
  -H "Content-Type: application/json" \
  -d '{"message_id":"12345","code":0}'
```

### 鉴权

如果配置了 `http_access_token`，请加上 Header：
```
Authorization: Bearer <token>
```
或 query：`?access_token=<token>`

## 路径 B：发消息时附带回应（`send_group_msg` / `send_private_msg`）

适用于"**回应按钮的同时发一条消息**"的场景，**一次 API 调用完成两件事**。

### 在 OneBot v11 action params 中追加：

| 字段 | 类型 | 说明 |
|---|---|---|
| `interaction_id` | string | 平台原始 `d.id`（即按钮回调事件 payload 的 `data.ID` 字段） |
| `interaction_code` | int | 回应代码 0~5，默认 `0`（成功） |

> 该路径目前仅识别 `interaction_id`，**不接受 `message_id` 短 ID**。如需用短 ID，请走路径 A。

### 群消息示例
```json
{
    "action": "send_group_msg",
    "params": {
        "group_id": "123456789",
        "message": "操作成功！",
        "interaction_id": "1234567890abcdef",
        "interaction_code": 0
    }
}
```

### 私聊消息示例
```json
{
    "action": "send_private_msg",
    "params": {
        "user_id": "987654321",
        "message": "权限不足！",
        "interaction_id": "abcdef1234567890",
        "interaction_code": 4
    }
}
```

### 仅回应、不发消息（不推荐，建议改用路径 A）
```json
{
    "action": "send_group_msg",
    "params": {
        "group_id": "123456789",
        "message": "",
        "interaction_id": "fedcba0987654321",
        "interaction_code": 0
    }
}
```

## 🪝 Webhook 模式特性（强烈推荐）

如果你以 **webhook 模式** 接收事件（即 QQ 平台向你的服务器主动 POST 推送回调），框架内置了一套 **"槽位机制"**，可以**绕过平台白名单受限的 PUT API**，直接在 webhook 200 OK 响应体里返回 `code`：

### 工作原理

1. webhook 收到 `INTERACTION_CREATE` 事件 → 框架为该 `interaction_id` 注册一个 pending 槽位
2. 应用端在 **`webhook_resp_wait_ms`** 毫秒内调用 `/put_interaction`（或 `send_group_msg` 携带 `interaction_id`）→ 槽位被填充
3. webhook 立即用应用端给的 `code` 拼成 `{"op":12,"code":N}` 响应平台 → **完全不走平台 PUT API**
4. 超时未填充 → 走 `webhook_resp_code` 兜底 code

### 关键配置项（`config.yml`）

```yaml
webhook_resp_wait_ms: 1000   # 等待应用端覆盖 code 的最大毫秒数。0=不等待立即用兜底。
webhook_resp_code:   0       # 应用端无覆盖时的兜底 code。0=操作成功（平台默认提示）
```

### 为什么这是优势路径？

- **无需邮件申请白名单**：平台原生 `PUT /interactions/{id}` 接口需要邮件申请，而 webhook 200 OK 响应体的 `op:12 + code` 是**公开能力**
- **更快**：少一次外部 HTTP 调用，直接在收到事件的 TCP 连接上回包
- **应用端不感知**：调用 `/put_interaction` 的接口签名完全一样，框架自动判断走槽位还是走平台 PUT

> 关闭槽位机制：把 `webhook_resp_wait_ms` 设为 `0`，框架立即用 `webhook_resp_code` 兜底，不等待应用端。

## � 完整工作流程

```
用户点击按钮
   ↓
QQ平台推送 INTERACTION_CREATE 到 webhook
   ↓
框架解析事件，注册 pending 槽位（仅 webhook 模式 + webhook_resp_wait_ms > 0）
   ↓
框架将事件转发给应用端（按 global_interaction_to_message 配置选择转 message 或 notice）
   ↓
应用端业务处理 → 决定 code
   ↓
应用端调用 /put_interaction 或 send_group_msg(interaction_id=..)
   ↓
框架优先填充 webhook 槽位 → webhook 用 {"op":12,"code":N} 应答平台
（如果不在 webhook 等待窗口内，回退调用平台 PUT /interactions/{id}）
```

## � 如何获取 `interaction_id`

按钮回调事件 payload 中的 **`d.id`** 字段就是 `interaction_id`：

```json
{
    "id": "EVENT_xxxxxxxxxxxx",       // ← 这是 event_id，不要用！
    "t": "INTERACTION_CREATE",
    "d": {
        "id": "1a2b3c4d5e6f7890",     // ← 这才是 interaction_id ✅
        "type": 11,
        "scene": "c2c",
        "group_openid": "GROUP_xxx",
        "data": {
            "resolved": {
                "button_id": "btn_001",
                "button_data": "confirm_action"
            }
        }
    }
}
```

如果开启了 `global_interaction_to_message: true`，框架在转给应用端的消息体上会自动追加 `interaction_id` 字段，详见 `按钮回调消息转换增强说明.md`。

## ⚠️ 注意事项

- 每个 `interaction_id` **只能回应一次**，重复回应会失败
- webhook 模式下，回应必须在 `webhook_resp_wait_ms` 毫秒内完成；超时则只能走平台 PUT API（需白名单）
- `code` 必须在 `0~5` 范围内，否则接口返回错误
- `auto_put_interaction: false` 时（推荐配置），所有回应必须手动触发

## 💻 应用端示例

### Python（webhook 模式 + `/put_interaction`）

```python
import requests

GSK_BASE = "http://127.0.0.1:5700"

def handle_button_callback(event):
    """按钮回调业务处理"""
    interaction_id = event["d"]["id"]
    button_data = event["d"]["data"]["resolved"]["button_data"]

    if button_data == "approve":
        code = 0   # 成功
    elif button_data == "deny":
        code = 4   # 没有权限
    else:
        code = 1   # 操作失败

    # 通过独立接口回应
    requests.post(f"{GSK_BASE}/put_interaction", json={
        "interaction_id": interaction_id,
        "code": code,
    })
```

### JavaScript（回应 + 发消息合并）

```javascript
async function handleButtonCallback(event) {
    const interactionId = event.d.id;
    const buttonData = event.d.data.resolved.button_data;

    let message, code;
    if (buttonData === "confirm_action") {
        message = "操作已确认！";
        code = 0;
    } else {
        message = "未知操作";
        code = 1;
    }

    // 一次 API 同时回应按钮 + 发送消息
    await fetch("http://127.0.0.1:5700/send_group_msg", {
        method: "POST",
        headers: { "Content-Type": "application/json" },
        body: JSON.stringify({
            group_id: event.d.group_openid,
            message: message,
            interaction_id: interactionId,
            interaction_code: code,
        }),
    });
}
```

## 🔍 日志关键字

便于排查问题，可在日志中搜索：

- **路径 A 命中槽位**：`put_interaction 通过webhook槽位回应交互: ID=..., Code=N`
- **路径 A 走平台 PUT**：`put_interaction` 无错误日志即成功
- **路径 B 命中槽位**：`发送群消息时通过webhook槽位回应交互: ID=..., Code=N`
- **webhook 应用端 code**：`[webhook] interaction ... 应用端 code=N`
- **webhook 兜底 code**：`[webhook] interaction ... 兜底 code=N`
- **message_id 还原失败**：`put_interaction message_id 还原失败: ...`
